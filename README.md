# PE64 Revenant Patcher

**PE64 Revenant** — исследовательская утилита для лабораторного анализа и модификации PE64/AMD64-исполняемых файлов Windows.

Скрипт добавляет или переиспользует секцию `.run`, записывает в нее минимальный x64 loader-stub, перенаправляет `AddressOfEntryPoint` на этот stub, вызывает `LoadLibraryW` для указанной DLL, после чего передает управление исходной точке входа программы.

Проект предназначен для изучения структуры PE-файлов, reverse engineering, DFIR-обучения, malware analysis lab, разработки детектов и контролируемых security research-сценариев на файлах, которые принадлежат исследователю или разрешены к модификации.

> **Важно:** проект не предназначен для несанкционированного изменения сторонних исполняемых файлов, обхода средств защиты, скрытой загрузки кода или эксплуатации чужих систем.

## Назначение

Скрипт демонстрирует, как программно модифицировать структуру PE64-файла:

- проверить, что файл является PE64/AMD64;
- найти импорт `LoadLibraryW` в Import Address Table;
- создать дополнительную PE-секцию `.run`;
- записать в нее x64 stub;
- изменить `AddressOfEntryPoint`;
- сохранить исходный OEP и вернуться к нему после выполнения stub;
- обновить `SizeOfImage`;
- выполнить базовую пост-проверку структуры итогового файла.

Основная прикладная ценность проекта — не эксплуатация, а изучение PE-структур и подготовка тестовых образцов для анализа, сигнатур, YARA/Sigma-логики, EDR-телеметрии и учебных DFIR-сценариев.

## Возможности

- Проверка PE-сигнатуры `MZ`.
- Проверка архитектуры AMD64.
- Проверка `OptionalHeader.Magic == PE32+`.
- Поиск `LoadLibraryW` в таблице импортов.
- Создание новой секции `.run`.
- Переиспользование существующей `.run` через `--keep-old-run`.
- Автоматическое расширение PE-заголовков при отсутствии места для нового `IMAGE_SECTION_HEADER`.
- Обновление:
  - `NumberOfSections`;
  - `AddressOfEntryPoint`;
  - `SizeOfImage`;
  - `SizeOfHeaders` при расширении заголовков;
  - `PointerToRawData` у сдвинутых секций;
  - смещения certificate table при расширении заголовков.
- Проверка пересечения секций по RVA и RAW.
- Обнаружение overlay-данных.
- Обнаружение Authenticode certificate table.
- Строгий режим проверки `--strict`.
- Диагностический отчет после успешного патчинга.

## Быстрый старт

Базовый запуск:

```bash
python pe64_Revenant_patcher.py --trusted input.exe --payload-dll C:\Lab\payload.dll --out patched.exe
```

Запуск со строгой предварительной проверкой:

```bash
python pe64_Revenant_patcher.py --trusted input.exe --payload-dll C:\Lab\payload.dll --out patched.exe --strict
```

Переиспользование существующей секции `.run`:

```bash
python pe64_Revenant_patcher.py --trusted input.exe --payload-dll C:\Lab\payload.dll --out patched.exe --keep-old-run
```

Рекомендуемый вариант для лабораторного применения:

```bash
python pe64_Revenant_patcher.py --trusted input.exe --payload-dll C:\Lab\payload.dll --out patched.exe --strict
```

## Аргументы командной строки

| Аргумент | Обязательный | Описание |
|---|---:|---|
| `--trusted` | Да | Путь к входному PE64/AMD64 EXE-файлу. Файл должен импортировать `LoadLibraryW`. |
| `--payload-dll` | Да | Путь к DLL, который будет записан в loader-stub и передан в `LoadLibraryW`. |
| `--out` | Да | Путь для сохранения модифицированного EXE-файла. Если расширение не `.exe`, скрипт автоматически заменит его на `.exe`. |
| `--strict` | Нет | Включает строгие pre-flight проверки. При нарушениях скрипт завершает работу без патчинга. |
| `--keep-old-run` | Нет | Разрешает переиспользовать уже существующую секцию `.run`, если она является последней по RVA и RAW. |

## Логика работы

### 1. Загрузка входных данных

Скрипт получает пути к входному EXE, DLL и выходному EXE. Затем выполняются базовые проверки:

- существует ли входной EXE;
- существует ли DLL;
- начинается ли входной файл с сигнатуры `MZ`;
- корректно ли файл разбирается библиотекой `pefile`.

Если выходной путь не заканчивается на `.exe`, расширение автоматически заменяется на `.exe`.

---

### 2. Проверка формата PE64/AMD64

Скрипт поддерживает только 64-битные PE-файлы Windows:

- `Machine == 0x8664`, то есть AMD64;
- `OptionalHeader.Magic == 0x20B`, то есть PE32+.

PE32/x86, ARM, ARM64, IA-64, ROM-образы и другие форматы не поддерживаются.

---

### 3. Строгая предварительная проверка

При использовании `--strict` выполняются дополнительные проверки. Любое нарушение считается критическим и приводит к остановке.

Проверяется:

- отсутствие пересечений секций по RVA;
- отсутствие пересечений секций по RAW-смещениям;
- корректность `FileAlignment`;
- корректность `SectionAlignment`;
- условие `FileAlignment <= SectionAlignment`;
- порядок секций по виртуальным адресам;
- отсутствие уже существующей секции `.run`;
- отсутствие overlay-данных;
- отсутствие Authenticode certificate table;
- наличие места для нового заголовка секции.

`--strict` полезен, если нужно исключить неявное повреждение структуры PE-файла и получить предсказуемый результат.

---

### 4. Поиск `LoadLibraryW` в IAT

Скрипт не создает новый импорт в Import Table. Вместо этого он ищет уже существующий импорт `LoadLibraryW`.

Логика:

1. Берется `ImageBase` из Optional Header.
2. Перебираются записи `DIRECTORY_ENTRY_IMPORT`.
3. Для каждой импортируемой функции проверяется имя.
4. Если найден `LoadLibraryW`, вычисляется RVA IAT-записи:

```text
IAT_RVA = imported_function_address - ImageBase
```

Если `LoadLibraryW` отсутствует, скрипт завершает работу с ошибкой.

Это важное ограничение: входной EXE должен уже импортировать `LoadLibraryW` напрямую. Если программа получает адрес функции динамически через `GetProcAddress`, импортирует только `LoadLibraryA` или вообще не содержит нужной записи в IAT, скрипт не сможет выполнить патчинг.

---

### 5. Формирование loader-stub

Loader-stub — небольшой x64-фрагмент кода, который выполняется перед исходной точкой входа программы.

Логика stub:

```asm
sub  rsp, 28h
lea  rcx, [rip + dll_path]
call qword ptr [rip + LoadLibraryW_IAT]
add  rsp, 28h
jmp  old_oep
; UTF-16LE DLL path
```

Назначение инструкций:

- `sub rsp, 28h` — выделение shadow space и выравнивание стека в соответствии с Microsoft x64 ABI;
- `lea rcx, [rip + dll_path]` — передача первого аргумента в `RCX`, то есть пути к DLL;
- `call qword ptr [rip + LoadLibraryW_IAT]` — вызов `LoadLibraryW` через найденную IAT-запись;
- `add rsp, 28h` — восстановление стека;
- `jmp old_oep` — переход к исходной точке входа EXE;
- строка пути DLL хранится после кода в UTF-16LE с завершающим `\x00\x00`.

В stub не используются абсолютные виртуальные адреса. Переходы и обращения строятся через `rel32`, что делает stub совместимым с ASLR в рамках адресного пространства самого образа.

---

### 6. Создание или переиспользование секции `.run`

Секция `.run` используется как контейнер для loader-stub и строки пути к DLL.

Имя секции:

```text
.run
```

Характеристики секции:

```text
0xE0000020
```

Это означает:

- содержит код;
- доступна на чтение;
- доступна на запись;
- доступна на исполнение.

Размеры секции выравниваются по `FileAlignment` и `SectionAlignment`. Минимальный размер выбирается как максимум из:

- размера loader-stub;
- `0x1000`;
- требований выравнивания.

Если `.run` уже существует, поведение зависит от режима:

- без `--keep-old-run` скрипт завершит работу с ошибкой;
- с `--keep-old-run` скрипт переиспользует `.run`, но только если она является последней секцией по RVA и RAW.

---

### 7. Расширение PE-заголовков

Если в заголовках нет места для нового `IMAGE_SECTION_HEADER`, скрипт может автоматически расширить область заголовков.

Механизм:

1. Перед RAW-данными первой секции вставляется блок нулей размером `FileAlignment`.
2. Увеличивается `SizeOfHeaders`.
3. Для секций, чьи RAW-данные были сдвинуты, обновляется `PointerToRawData`.
4. Если в файле есть certificate table и она находится после сдвинутой области, обновляется ее file offset.

В режиме `--strict` автоматическое расширение не выполняется. Если места для нового заголовка нет, скрипт завершает работу с ошибкой.

---

### 8. Обработка overlay и certificate table

Скрипт умеет определять:

- overlay-данные;
- Authenticode certificate table.

Особенность PE-формата: `DATA_DIRECTORY[SECURITY].VirtualAddress` для certificate table является файловым смещением, а не RVA. Скрипт учитывает это отдельно.

Поведение:

- если новая секция пересекает certificate table, скрипт завершает работу;
- если новая секция пересекает overlay:
  - в `--strict` режиме скрипт завершает работу;
  - без `--strict` выводится предупреждение, overlay может быть перезаписан.

Важно: любая модификация подписанного PE-файла делает Authenticode-подпись недействительной. Даже если certificate table физически не повреждена, содержимое файла уже изменено, поэтому исходная подпись больше не должна считаться валидной.

---

### 9. Патчинг Optional Header

После записи stub скрипт обновляет Optional Header:

- `AddressOfEntryPoint` заменяется на RVA секции `.run`;
- `SizeOfImage` увеличивается, если новая секция выходит за пределы старого значения;
- `CheckSum` сбрасывается в `0`.

Исходный OEP сохраняется внутри stub как цель финального перехода.

---

### 10. Запись результата

После подготовки секции и заголовков скрипт:

1. расширяет файл нулями до требуемого размера;
2. очищает RAW-область `.run`;
3. записывает loader-stub в начало `.run`;
4. сохраняет измененный EXE по пути `--out`.

---

### 11. Пост-проверка результата

После записи выходного файла скрипт повторно открывает его через `pefile` и проверяет:

- наличие сигнатуры `MZ`;
- сохранение архитектуры AMD64;
- сохранение формата PE32+;
- совпадение `NumberOfSections` с количеством разобранных секций;
- отсутствие пересечений секций;
- что `AddressOfEntryPoint` не равен нулю;
- что `AddressOfEntryPoint` указывает на ожидаемый RVA;
- что секция `.run` существует;
- что `.run` имеет ненулевой RAW- и виртуальный размер;
- что `.run` имеет флаг исполнения;
- что новый Entry Point находится внутри `.run`;
- что `SizeOfImage` не меньше расчетного значения;
- что Import Table читается после модификации.

Если одна из проверок не проходит, скрипт завершает работу с ошибкой.

---

## Что скрипт не делает

Скрипт не выполняет следующие действия:

- не внедряется в уже запущенные процессы;
- не выполняет process injection;
- не выполняет reflective loading;
- не создает persistence-механизмы;
- не обходит EDR/AV;
- не маскирует изменения в PE-файле;
- не восстанавливает и не пересчитывает Authenticode-подпись;
- не добавляет новый импорт в Import Table;
- не исправляет CFG/Load Config Directory;
- не гарантирует совместимость с packed/protected binaries;
- не гарантирует корректную работу с .NET/managed PE;
- не проверяет логику самой DLL;
- не проверяет наличие зависимостей у DLL.

## Ограничения

### Архитектура

Поддерживается только:

```text
Machine: AMD64 / 0x8664
OptionalHeader.Magic: PE32+ / 0x20B
```

Не поддерживаются:

- PE32/x86;
- ARM;
- ARM64;
- IA-64;
- драйверы со специфичными требованиями загрузки;
- ROM-образы;
- нестандартные или поврежденные PE-файлы.

---

### Импорты

Входной EXE должен уже импортировать `LoadLibraryW`.

Если импорт отсутствует, скрипт завершится с ошибкой:

```text
[!] LoadLibraryW not found in IAT.
```

Скрипт не добавляет `LoadLibraryW` в Import Table самостоятельно.

---

### Путь к DLL

Путь к DLL записывается в stub как абсолютный путь.

Скрипт использует:

```python
Path(args.payload_dll).resolve()
```

Поэтому рекомендуется запускать патчер на Windows и передавать валидный Windows-путь, например:

```text
C:\Lab\payload.dll
```

Если запустить скрипт на Linux/macOS, в stub может попасть путь формата `/home/user/payload.dll`, который не будет валидным путем для Windows.

---

### Разрядность DLL

DLL должна соответствовать архитектуре процесса. Для PE64/AMD64 EXE нужна 64-битная DLL.

32-битная DLL не загрузится в 64-битный процесс.

---

### Цифровая подпись

После модификации PE-файла цифровая подпись становится недействительной.

Даже если certificate table не была физически перезаписана, изменение `AddressOfEntryPoint`, добавление секции и изменение заголовков нарушают целостность подписанного файла.

---

### Overlay

Overlay — данные, находящиеся после последней секции PE-файла.

Некоторые программы используют overlay для:

- установочных данных;
- ресурсов;
- архивов;
- самораспаковки;
- служебных метаданных;
- защитных механизмов.

В режиме без `--strict` overlay может быть перезаписан. Поэтому для аккуратной работы с неизвестными файлами рекомендуется использовать `--strict`.

---

### Существующая секция `.run`

Если `.run` уже существует, скрипт по умолчанию завершит работу.

Переиспользование разрешается только через:

```bash
--keep-old-run
```

При этом `.run` должна быть последней секцией:

- по RVA;
- по RAW-смещению.

Если это условие не выполняется, переиспользование считается небезопасным.

---

### Packed/protected binaries

Скрипт не предназначен для работы с packed/protected binaries. Упаковщики и защитные оболочки могут:

- проверять целостность PE-заголовков;
- контролировать исходный Entry Point;
- использовать overlay;
- хранить данные после секций;
- изменять порядок секций;
- выполнять собственную распаковку до передачи управления.

Для таких файлов результат не гарантируется.

---

## Режим `--strict`

Режим `--strict` рекомендуется использовать для первичной проверки входного файла.

Он запрещает:

- overlay;
- certificate table;
- существующую `.run`;
- неупорядоченные секции;
- пересечения RVA/RAW;
- нестандартные alignment-значения;
- автоматическое расширение заголовков при нехватке места.

Пример:

```bash
python pe64_Revenant_patcher.py --trusted input.exe --payload-dll C:\Lab\payload.dll --out patched.exe --strict
```

Если strict-проверка не проходит, это не всегда означает, что файл нельзя модифицировать. Это означает, что автоматическая модификация может повредить структуру файла или привести к неоднозначному результату.

После успешного выполнения скрипт выводит отчет:

- Machine;
- Magic;
- FileAlignment;
- SectionAlignment;
- старый Entry Point;
- новый Entry Point;
- список секций;
- RVA секций;
- RAW-смещения;
- размеры секций;
- характеристики секций;
- сведения об overlay;
- сведения о certificate table;
- расчетный `SizeOfImage`;
- финальный `SizeOfImage`.

Также выводятся:

- путь к входному EXE;
- путь к DLL;
- путь к выходному EXE;
- RVA старого OEP;
- RVA нового OEP;
- RVA `LoadLibraryW`;
- была ли добавлена `.run`;
- RVA и RAW `.run`;
- размер `.run`;
- характеристики `.run`.

## Типовые ошибки

### `Not PE64`

Причина: входной файл не является PE64/AMD64.

Решение: использовать 64-битный EXE-файл Windows.

---

### `LoadLibraryW not found in IAT`

Причина: входной EXE не импортирует `LoadLibraryW`.

Решение: использовать другой лабораторный EXE, который содержит прямой импорт `LoadLibraryW`.

---

### `.run already exists`

Причина: входной файл уже содержит секцию `.run`.

Решение: проверить файл вручную. Если секция точно предназначена для переиспользования и является последней по RVA и RAW, можно использовать `--keep-old-run`.

---

### `STRICT: no room for new section header`

Причина: в PE-заголовках нет свободного места для нового `IMAGE_SECTION_HEADER`, а strict-режим запрещает автоматическое расширение заголовков.

Решение: запустить без `--strict` только в лабораторной среде или подготовить файл вручную в PE-редакторе.

---

### `overlay will be overwritten`

Причина: новая `.run` пересекается с overlay-данными.

Решение: использовать `--strict`, проверить overlay вручную и не модифицировать файл, если overlay важен для работы программы.

---

### `certificate table`

Причина: файл содержит Authenticode certificate table.

Решение: не модифицировать подписанный файл, если требуется сохранить подпись. Для лабораторного анализа используйте неподписанный тестовый образец.

---

## Рекомендованный лабораторный workflow

1. Использовать только тестовую VM.
2. Использовать только собственные или специально подготовленные PE-файлы.
3. Сделать копию исходного EXE.
4. Зафиксировать хэш исходного файла.
5. Запустить скрипт с `--strict`.
6. Проанализировать диагностический отчет.
7. Зафиксировать хэш выходного файла.
8. Проверить структуру результата в PE-редакторе.

## DFIR и detection engineering: полезные признаки

Модифицированный файл может быть полезен как учебный образец для проверки детектов на изменение PE-структуры.

Потенциальные признаки для анализа:

- `AddressOfEntryPoint` указывает в нестандартную новую секцию;
- появилась секция `.run`;
- секция имеет флаги `READ | WRITE | EXECUTE`;
- изменилось количество секций;
- изменился `SizeOfImage`;
- `CheckSum` сброшен;
- Authenticode-подпись стала недействительной;
- в новой секции присутствует UTF-16LE-строка с путем к DLL;
- Entry Point находится не в исходной `.text`, а в добавленной секции;
- наблюдается переход из новой секции обратно к исходному OEP.

Эти признаки не следует считать самостоятельным доказательством вредоносности. Они указывают на модификацию структуры PE-файла и требуют контекстного анализа.
